## Problem Statement

The extension should feel like a professional database workbench inside VS Code, not a collection of isolated database commands. The target learning workflow is:

```text
connect -> browse schema -> write SQL -> run section/query -> inspect results -> reuse history
```

The current code already has a useful foundation: PostgreSQL/Redshift drivers, stored connections, a database tree, SQL statement splitting, statement CodeLens, basic metadata completion, query execution, query history, and a React results grid. The next step is to connect those pieces into a cohesive IDE experience.

## What Makes This Cool

The "whoa" moment is opening VS Code, choosing a saved connection, seeing schema context immediately, writing SQL with real table/column suggestions, running exactly the current query section, and being able to recover every useful query later even after result tabs close.

AI should make the learning loop faster:

- Explain this query in plain language.
- Fix this SQL error using the current schema.
- Generate a starter query from selected tables.
- Summarize the result set and suggest the next question to ask.

AI is secondary to the workbench. If autocomplete, history, section execution, and results are weak, AI will feel bolted on.

## Constraints

- Keep the UI VS Code-native: dense, predictable, keyboard-friendly, and theme-aware.
- Avoid decorative redesign. Professional means clear state, strong hierarchy, and fast repeated use.
- Reuse the current drivers and result grid unless a specific limitation blocks the workbench.
- Do not store passwords outside VS Code secrets.
- Treat SQL execution safety as product scope, especially for destructive queries and production connections.
- Preserve learning value: expose why a suggestion or AI answer is generated instead of hiding everything behind magic.

## Premises

1. The core product is the full workflow, not one feature: connect, browse, write, run, inspect, reuse.
2. A professional database UI should be dense and IDE-native, not a marketing-style web app.
3. Autocomplete must use schema context, aliases, and query position to be useful.
4. Query history must outlive result tabs, extension reloads, and editor closure.
5. AI features should help users understand, generate, repair, and explore SQL without replacing the fundamentals.

## Approaches Considered

### Approach A: Polish Current Foundation

Improve the current connection prompts, autocomplete, query history, CodeLens sections, and results UI in place.

Effort: M
Risk: Low

Pros:

- Fastest path to visible improvement.
- Reuses nearly all existing modules.
- Lower chance of destabilizing execution and results behavior.

Cons:

- Connection setup remains limited by step-by-step VS Code input boxes.
- Autocomplete likely stays shallow without a dedicated schema context layer.
- AI features risk feeling like unrelated commands.

Reuses:

- `ConnectionManager`
- `ConnectionStore`
- `splitSqlStatements`
- `QueryExecutor`
- `ResultsPanelProvider`
- Results React app

### Approach B: Cohesive Database Workbench

Build a cohesive workbench layer: professional connection editor, persistent query consoles/files, schema introspection cache, smart autocomplete provider, query section navigator, durable query history, and AI SQL assistant commands.

Effort: L
Risk: Medium

Pros:

- Matches the stated "whole flow" goal.
- Creates a real product shape instead of scattered improvements.
- Builds reusable architecture for future database features.

Cons:

- Touches more files and needs stronger tests.
- Requires careful state design for active connection, schema cache, history, and result tabs.
- AI features need provider boundaries and safe context handling.

Reuses:

- Existing database drivers.
- Existing explorer provider and node model.
- Existing results grid.
- Existing query history store.
- Existing SQL splitter and execution path.

### Approach C: AI-First SQL Coach

Lead with AI explanations, query review, result interpretation, and "fix this SQL" flows while only lightly improving UI/autocomplete.

Effort: M
Risk: Medium

Pros:

- Strong learning value quickly.
- More differentiated than another database explorer.
- Demo-friendly.

Cons:

- Does not solve the full workbench fundamentals.
- AI quality depends on schema context and provider setup.
- Can hide weak product architecture instead of fixing it.

Reuses:

- Query executor.
- Query history.
- Schema metadata calls.
- VS Code command palette.

## Recommended Approach

Choose Approach B: Cohesive Database Workbench.

This is the right fit because the user's goal is the whole flow. The extension should feel professional from connection setup through query reuse. That requires a workbench architecture, not isolated polish.

## Workbench Architecture

```text
VS Code Commands / Views
  |
  +-- Connection Workbench Webview
  |     +-- create/edit/test/delete profiles
  |     +-- production/read-only/default schema flags
  |
  +-- Database Explorer
  |     +-- schemas/tables/views/columns/indexes/keys
  |     +-- context actions generate/open/query
  |
  +-- SQL Editor Integration
  |     +-- connection binding per document
  |     +-- section detection and navigator
  |     +-- schema-aware completions
  |     +-- AI assistant commands
  |
  +-- Results Panel
  |     +-- result tabs
  |     +-- multiple result sets
  |     +-- filters/export/inspector
  |
  +-- Persistence Layer
        +-- connection metadata + secrets
        +-- query history
        +-- query consoles/files
        +-- schema cache
```

## Proposed Feature Scope

### 1. Professional Connection UI

Replace the sequential input-box flow with a single connection editor webview.

Fields:

- Name
- Database type
- Host
- Port
- Database
- Username
- Password
- SSL mode
- Default schema
- Connection color
- Query timeout
- Production flag
- Read-only default

States:

- Empty: no connections yet, focused "Add connection" panel.
- Editing: form with validation and test button.
- Testing: spinner and disabled save.
- Success: server version and latency.
- Error: exact connection error with host/port/user context.

### 2. Persistent Query Consoles

Create a query console concept separate from result tabs.

Each console should remember:

- Connection id
- Database/schema context
- Untitled or saved SQL document URI
- Last executed statement range
- Created/updated timestamps

Expected behavior:

- Opening a connection can create or reopen its console.
- Closing a result tab does not lose the SQL.
- Query history remains searchable separately.
- Users can pin useful consoles or save them as `.sql` files.

### 3. Smart SQL Sections

Upgrade current statement CodeLens into a section-aware query experience.

Section behavior:

- Parse SQL into statements with stable ranges.
- Show "Run" above each statement.
- Add a query section outline command.
- Highlight only the running section.
- Preserve result tab source range.

Important edge cases:

- Semicolons inside strings.
- Dollar-quoted PostgreSQL functions.
- Comments between statements.
- Empty sections.
- Selection-as-single-statement execution.

### 4. Schema-Aware Autocomplete

Move autocomplete from direct driver calls to a schema context service.

Completion layers:

- SQL keywords.
- Schemas.
- Tables/views.
- Columns after table or alias.
- Functions and snippets.
- Generated joins from foreign keys.

Context examples:

- `from |` suggests tables/views.
- `join |` suggests tables/views.
- `select u.|` resolves alias `u` to a table and suggests columns.
- `where |` suggests columns from tables already in scope.

The schema cache should have a TTL, manual refresh, and per-connection invalidation.

### 5. Durable Query History

Query history should be a first-class tool, not a byproduct of execution.

History fields:

- SQL text
- Connection
- Database type
- Source file
- Source range
- Status
- Error message
- Row count
- Duration
- Executed timestamp
- Favorite/star flag

History UI:

- Search by SQL text.
- Filter by connection/status/date.
- Reopen in console.
- Copy SQL.
- Favorite.
- Delete individual entry.

### 6. Results Workbench Polish

Keep the current React grid but tighten the professional feel.

Improvements:

- Use icon-first toolbar buttons with tooltips.
- Keep result-set tabs compact and stable.
- Show execution messages/notices next to grids.
- Add clear empty/error/partial states.
- Preserve sort/filter/column state per result tab.
- Make row limits explicit before rerun.

### 7. AI SQL Assistant

Add AI commands only after schema context and history are reliable.

Commands:

- Explain selected query.
- Fix last SQL error.
- Generate query from selected schema/table context.
- Optimize query with caveats.
- Summarize result set.

AI context boundaries:

- Send selected SQL.
- Send relevant schema names, table names, column names.
- Send last error details.
- Do not send passwords or connection secrets.
- Let users choose provider/API key later.

AI UI should show generated SQL in an editor diff or insert preview, never silently execute it.

## Data Flow

```text
User edits SQL
  |
  v

  |-- detects current statement
  |-- maps aliases and tables
  |
  v

```

## Open Questions

- Which AI provider should be supported first: OpenAI API, VS Code Language Model API, local Ollama, or a provider-agnostic adapter?
- Should query consoles be backed by real files in the workspace, VS Code untitled documents, or extension-owned virtual documents?
- Should destructive SQL confirmation apply only to production connections, or optionally to all connections?
- Should autocomplete introspect all schemas by default or only the default schema until users expand more?

## Success Criteria

- A new user can add a connection, test it, and open a query console without using command palette commands.
- SQL autocomplete suggests real tables and columns for the active connection.
- Running a section executes the intended statement and highlights the exact range.
- Closing a result tab does not remove the query from durable history.
- Query history can reopen a prior query after extension reload.
- AI commands never include secrets and never execute generated SQL automatically.
- The extension remains theme-aware and usable in light/dark VS Code themes.

## Distribution Plan

The deliverable is a VS Code extension. Existing packaging is defined by `package.json` and the compiled entrypoint is `dist/extension.js`.

Needed before public distribution:

- Add a repeatable build/test command.
- Add CI for typecheck/build.
- Add extension packaging using `vsce` or equivalent.
- Decide whether generated `dist/` remains committed or is built during packaging.
